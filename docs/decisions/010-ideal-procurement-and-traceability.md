# ADR 010: [Ideal Concept] 資材調達・在庫トレーサビリティの深層データモデル設計方針

## ステータス
**提案中 (Proposed)**: 2026-04-07
※ 本件はフェーズ2以降の「理想系」を見据えた構想であり、実装時に要件に応じて変更される可能性があります。

## 背景 (Context)
製造現場における資材調達は、単なる「買って届く」だけではなく、以下の複雑なパターンへの対応が求められる。
- **分納とまとめ請求**: 1つの発注が3回に分かれて届き、支払いは翌月に他の発注とまとめて1つの請求書で行われる。
- **代理入力と実要求者の分離**: 現場の忙しい担当者に代わって事務員が入力するが、問い合わせや予算責任は現場担当にある。
- **多様な取引形態**: 通常購入に加え、自社支給（外注加工）や客先支給（無償受領）が混在し、在庫の増減ロジックは同じだが会計上の扱いが異なる。

## 決定事項 (Decision) - [Ideal Concept]

### 1. ヘッダ・明細構造の徹底と追跡チェーンの確立
資材のライフサイクルを以下のテーブルチェーンで管理し、各段階で「多対多」の紐付けを許容する。
- **資材要求 (`material_requests`)** -> **発注 (`purchase_orders`)** -> **入荷 (`receipt_lines`)** -> **仕入請求 (`supplier_invoice_lines`)**
- 紐付け専用の交差テーブル（`material_request_fulfillments`）を介することで、どの要求がどの発注で満たされたかを厳密に追跡する。

### 2. トランザクション・アクターの役割定義
[ADR 009](./009-role-based-acl-and-representation.md) の思想に基づき、伝票上の人物を以下のように定義する。
- **`created_by_id`**: システムへのデータ入力者。
- **`applicant_id`**: 実際に資材を必要としている実要求者（表示担当者）。
- 理由：実務上の「代理入力」を許容しつつ、責任の所在を明確にするため。

### 3. 取引区分 (`transaction_type`) によるロジック分岐
伝票ヘッダに `transaction_type` を持たせ、以下の挙動を制御する。
- `1: 購入`: 標準フロー（請求あり）。
- `2: 自社支給`: 在庫は減るが、売上や仕入は発生しない（外注先への送付）。
- `3: 客先支給`: 在庫は増えるが、支払は発生しない（無償受領）。

## 期待される効果 (Consequences)
- **監査耐性の向上**: 1枚の請求書から、元となった複数の入荷、さらにその元の要求までをドリルダウンして調査できる。
- **現場の利便性**: 分納や一部返品、代理申請などの「例外的な日常業務」をシステムが標準機能として吸収できる。
- **在庫精度の向上**: 支給品が「なぜ増減したか」を通常の購入と区別して正確に把握できる。
