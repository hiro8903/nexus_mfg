# ADR 005: フロントエンド技術選定 (Hotwire + Tailwind CSS)

## 📅 ステータス (Status)
承認済み (Accepted): 2026-03-26（現状のドラフト版リファクタリング方針として更新）

## 🕯️ 背景 (Background)
現在、`app/views/users/` の主要なビューはドラフト段階であり、暫定的な HTML 構造やインラインスタイルが残存している。これを機に、アプリ全体のフロントエンド品質を統一し、保守性を最大化するための「標準的な作り方」を定義する必要がある。

すでに Hotwire が導入されているため、これを最大限に活かしつつ、モダンな UI デザインを実現する。

## 🗣️ 機能要求 (Functional Requirements)
- **サクサク動く操作感**: Hotwire による高速な画面推移と非同期処理。
- **デザインの統一性**: Tailwind CSS によるコンポーネント指向なスタイリング。
- **拡張性**: 複雑な動的機能が必要になった場合、Vue.js や React を「必要な箇所だけ」スムーズに導入できること。

## 🎯 意思決定 (Decision)
**Hotwire (Turbo/Stimulus) と Tailwind CSS を標準スタックとし、Rails のコンポーネント指向（Partial/ViewComponent）で UI を構築する。**

1.  **CSS**: `tailwindcss-rails` を採用し、TIL の標準ガイドに従って実装する。
2.  **HTML**: Rails パーシャルを用いて UI を共通化し、「同じ見た目・同じ作り方」を徹底する。
3.  **JS フレームワーク**: 原則として Hotwire で完結させる。ただし、後述の「高度な動的機能」が必要になった場合に限り、Vue.js や React を検討する。

## 💡 JS フレームワーク（React/Vue）導入の判断基準
以下の 4 つのような「Excel 以上の高度なインタラクティブ性」が求められたタイミングが、導入検討の基準（マイルストーン）となる。

1.  **組織図のビジュアル編集**: ドラッグ＆ドロップによる組織階層の組み換えと、権限のリアルタイム再計算。
2.  **在庫マップの可視化**: 工場や倉庫の平面図（SVG）と在庫情報の動的な連動。
3.  **タイムライン形式の配属管理**: カレンダーやガントチャートを用いた期間調整の視覚的操作。
4.  **権限継承のシミュレーター**: 大規模なマトリクス表における、データの波及効果（継承）のリアルタイム可視化。

**これらの機能が具体的に必要になるまでは、Hotwire + Tailwind CSS での実装を最優先する。**

## ⚖️ 意思決定の理由 (Rationale)
1.  **開発スピード**: Rails 標準の Hotwire は API 構築の手間を省き、サーバー主導で高速にプロ機能を提供できる。
2.  **保守の簡素化**: Tailwind のユーティリティクラスをコンポーネント（部品）に封じ込めることで、大規模化しても CSS が破綻しない。
3.  **将来への投資**: Import Maps 構成により、後から Vue や React を「特定のページ」だけに差し込むことが容易であり、技術的な袋小路に陥らない。

## 🌊 影響 (Consequences)
- 現在のドラフト版（`users/*.erb` 等）のインラインスタイルを、Tailwind クラスおよび共通パーシャルへ順次リファクタリングする。
- 新たに作成するビューは、全てこの標準スタックと TIL のガイドラインを遵守する。
