# ADR 002: 人事・組織構造および履歴管理の設計方針
**設計判断の記録 (Architecture Decision Record)**

## 📅 ステータス (Status)
**承認済み (Accepted)**: 2026-03-24

## 💡 背景 (Context)
製造業の中核システム（Manufacturing Core）において、「いつ、誰が、どの立場で、どの拠点で」業務や承認を行ったかという**トレーサビリティ（追跡可能性）**と、人事異動や組織改正（拠点の新設・廃止、部門の統廃合）に耐えうる堅牢なデータ構造が必要である。

## 🏹 検討した選択肢と要望 (Options & Requirements)

### 1. 履歴管理の設計
- **要望**: 部門や役職は変動が多く、過去の製造実績を確認する際に「その当時の権限・所属」を正確に記録・復元したい。
- **案A (シンプル)**: `User` モデルに `department_id` 等を直接持たせる。
- **案B (履歴重視)**: `assignments` テーブルを新設し、開始日・終了日で所属を管理する。
- **決定**: **案B (履歴重視)**
- **理由**: 会計監査や品質管理（バリデーション）において、過去の配属状態を再現できる拡張性を優先したため。

### 2. 名称の定義：拠点 (Site) と在庫場所 (Location)
- **要望**: 当初 (v1.0) は「ロケーション（棚・ラック等）」のみを想定していたが、今回の要件定義により「実際に人が勤務する拠点（本社・工場等）」も管理したいという要望が発生した。
- **課題**: `locations` という単一の言葉では、「物理的な棚」と「人事上の勤務地」の区別がつかず、混乱を招くリスクがある。

#### 比較検討した名称 (Workplace / Site)
| 候補 | ニュアンス | 評価 |
|:---|:---|:---|
| `sites` | 短くて一般的（Webサイト、土地等）。 | △ 製造現場の「施設・建物」という重みが薄い。 |
| **`facilities`** | **「設備・施設・工場」**。 | ✅ **採用。** 製造業ドメインに最も即した。専門的語彙。 |

#### 比較検討した名称 (Inventory Spot / Okiba)
| 候補 | ニュアンス | 評価 |
|:---|:---|:---|
| `storage_locations` | 正確だが長い。 | △ 可読性の低下。 |
| `storage_areas` | 「置き場」のイメージ（論理・物理両面）。 | ○ 直感的だが長い。 |
| `bins` | 棚の最小区画。 | △ 小分けの印象が強い。将来の拡張用に残す。 |
| **`locations`** | **業界標準（SAP等）の「置き場」。** | ✅ **採用。** 短く、かつ在庫管理の文脈で 100% 通じる。 |

- **結論**: **人事拠点を `facilities`、在庫置き場を `locations` とする。**
- **理由**: 拠点を `facilities` と明確化したことで、在庫管理シーンで最も一般的かつ簡潔な `locations` を本来（v1.0）の目的である「在庫の場所」として再定義する余地が生まれたため。

### 3. 名称の定義：「肩書き」を何と呼ぶか
- **検討した名称**: `position` vs `job_title`
- **比較考察**:
    - **`position`**: 一般的だが、製造現場では「位置（座標）」や「工程内の立ち位置」などの物理的な意味と混同されるリスクがある。
    - **`job_title`**: **「役職・名称」** であることが明確。人事システム（HRIS）の国際的標準でもあり、誤解の余地がない。
- **決定**: **`job_title`**
- **理由**: `position` は製造現場における「位置・座標」と混同されるリスクがあるが、`job_title` は「役職・名称」であることが直感的であり、人事システムの標準でもあるため。

### 4. 組織構造の設計：階層 (部・課・班) と横断組織 (PJ・委員会)
- **要望**: 部・課・班という深い階層構造に加え、特定のプロジェクトや委員会への「兼務」も表現したい。
- **検討**:
    - **案A (個別テーブル)**: `departments`, `sections`, `groups` などのテーブルを分ける。
    - **案B (統合テーブル)**: `org_units` ひとつにまとめ、`parent_id` (自己参照) で階層を表現する。
- **決定**: **案B (統合テーブル：`org_units`)**
- **理由**:
    1. **柔軟性**: 組織の階層がどれほど深くなっても、プログラム(Rails Model)を増やすことなく対応可能。
    2. **汎用性**: `org_type` カラム（部署、PJ、委員会）を持つことで、プロジェクトへの期間限定所属も `assignments` を介して一元的に管理できる。
    3. **命名のメリット**: `organizations` は他社サービスやライブラリの予約語（マルチテナント管理など）と衝突しやすいため、より具体的に「組織の最小単位」を指す `org_units` を採用した。

### 5. システム権限の管理：カラム名の検討（`system_role` vs `role` vs `job_title`）

> ⚠️ **この設計判断は ADR 003 にて再検討・変更されました。**
> 当初 `users.system_role` (integer/enum) をユーザーモデルに直接持たせる案を検討しましたが、
> 兼務時の権限合算や多様な権限粒度への対応を考慮した結果、
> `org_unit_permissions` テーブルによる動的継承モデルへ移行しています。
> **詳細は [ADR 003: 権限管理設計](./003-permission-management.md) を参照してください。**

- **当初の背景**: `User` モデルに「アプリケーション全体のシステム管理権限（管理者か否か）」を持たせる必要があるとして、以下を検討した。
- **検討した名称と棄却理由**:
  - **`job_title`**: 「役職名称の文字列（例：工場長、主任）」を意味する。ADR 項目 3 にて確定済み。肩書の名称であって、権限区分には不適切。
  - **`role`**: 一般的な Rails プロジェクトでは `job_title` が肩書を担うため、「`role` = システム権限」と読むのが一般的な解釈。しかし本プロジェクトでは `assignments.role`（**配属先での役職区分**）が既に同名カラムとして存在するため、`users.role` を追加すると「テーブルが違えば `role` の意味も違う」という文脈依存の混乱を招く。
  - **`system_role`**: `assignments.role`（配属役職）との区別が名前だけで明確になる。将来的な権限段階の拡張にも `enum` で柔軟に対応可能。
- **当初の決定（後にADR 003で変更）**: `system_role` (integer/enum) を `users` テーブルに持たせる。

### 🛠️ 自前実装を行う領域 (Custom Implementation)
- **人事配属・辞令ロジック (`assignments`)**:
  - 「誰がどこに所属しているか」という履歴自体は、システム要件の核（Business Core）であるため、外部ツールに依存せず、今回設計した `assignments` テーブルを用いて独自にコントローラーとサービスを実装する。
- **権限管理 (`org_unit_permissions`)**:
  - 「どの組織がどの権限を持つか」という定義も Business Core であるため、独自テーブルで管理する（詳細は ADR 003 参照）。
- **ハイブリッド在庫管理**:
  - 物理場所（Location）と論理所有組織（OrgUnit）を両立させる在庫管理ロジックは、既製のツールでは対応が難しいため、独自に構築する。

## ✅ 決定事項 (Decisions)
1.  **Identity と Assignment の分離**: ユーザーの「本人情報」と「配属履歴」を分離。
2.  **自己参照による組織階層**: `org_units.parent_id` による無制限の階層構造化（Ancestry を併用も視野に入れる）。
3.  **ハイブリッド在庫管理**: `inventories` において `org_unit_id` (所有：必須) と `location_id` (場所：任意) を共存させる。
4.  **外部ライブラリの積極活用**: 権限・変更履歴・階層などの再利用可能な機能は既存 Gem を活用するか、後日検討する。
5.  **動的権限継承モデルの採用**: `org_unit_permissions` テーブルを介して、配属先の組織から権限を自動付与する（詳細は ADR 003 参照）。

## 📈 期待される効果 (Consequences)
- 組織改正や拠点移動が発生しても、過去のデータを改ざんすることなく新しい情報を「時系列」に追加できる。
- `facilities` と `locations` を分離したことで、人事データと物流データのスキーマ衝突が回避され、インターフェースが劇的に分かりやすくなった。
- `inventories` のハイブリッド設計は、将来的な拡張において物理・論理どちらの側面からも在庫追記を可能にする。
- `org_units` の統一により、システム全体の「組織に関わるロジック」が共通化され、開発コストが削減される。
- 権限管理を組織に紐付けたことで、人事異動に伴う手動の設定変更コストが大幅に削減される。
