# ADR 008: 製造リソースの包括的管理のための拠点名称変更 (Facility to Site)

## ステータス
**承認済み (Accepted)**: 2026-04-06 (ADR 001/002 における以前の命名を上書き)

## 背景 (Context)
開発の初期段階において、工場や事務所などの物理的な拠点単位を `Facility` と命名していた。しかし、製造業のドメインモデルを詳細に検討した結果、将来的に導入される「設備（Equipment）」や「機械（Machinery）」、あるいはメンテナンス対象としての「施設（Facilities）」と、名称が衝突するリスクが極めて高いことが判明した。

長期的なアーキテクチャの透明性を確保するためには、「物理的な場所・地理的な拠点ユニット」を指す用語として、資産としての「設備」とは明確に区別できる用語を再定義する必要があった。

## 決定事項 (Decision)
1. **呼称の変更**: プロジェクト全域において `Facility` を `Site` に変更する。`Site` は、**SAP**（世界シェア1位を誇るドイツ発の基幹業務パッケージ）や **Oracle**（データベース技術を背景に強固な統合業務アプリを提供するアメリカ発のIT大手）などのグローバルなERPシステムにおいても、「物理的な事業拠点」を指す概念として広く認知されている。
2. **ドメインの分離とマイグレーションの分割**: 旧 `PersonnelBase` に混在していた「人・組織層」「物理拠点・ロケーション層」「在庫・資材層」を、ドメイン駆動設計の観点から 3 つのマイグレーション（`PersonnelBase`, `LogisticsBase`, `InventoryFoundation`）に論理的に分割・再編する。
3. **ハイブリッド在庫管理の明示**: 在庫テーブル（`Inventories`）において、「論理的な管理責任（`OrgUnit`）」と「物理的な保管場所（`Location`）」を完全に切り離し、同一の場所（`Location`）において複数部門（`OrgUnit`）が在庫を混在して所持できる「ハイブリッド型在庫管理」を基盤として確立する。
4. **歴史の遡及的修正 (Migrationのリセット)**: まだ開発の初期段階であり、データベース環境の所有権を開発チームが完全に保持していることを活かし、既存のマイグレーションファイルを直接書き換え、最初から「名前と中身が一致した」純粋なドメインごとの構成であったものとして再構築する。
5. **データ属性の拡充**: このリセットの機会を利用し、請求書の発行や納品指示書の印字に必須となる拠点詳細情報（`postal_code`, `phone_number`, `fax_number`）を `Site` モデルに追加し、データをリッチ化する。
6. **ユーザーアイデンティティのタクソノミー定義**: `Users` テーブルに `user_category`（社内/外/サービス）と `employment_type`（正社員/派遣等）を追加。ドメイン横断的な「人間・外部・自動化」のアイデンティティを、認証基盤の最深部で厳密に識別・隔離可能とする。

## 影響 (Consequences)
- **メリット**:
    - 将来的な「設備・資産マスタ（Equipment）」とのネームスペース衝突を完全に回避できる。
    - ドメイン（人事・物流・資産）ごとにファイルが分かれたことで、特定の業務ロジックの変更が他へ波及しにくくなり、保守性が劇的に向上した。
    - ハイブリッド在庫管理をマスタレベルで定義したことで、複雑な委託在庫や共通保管スペースの管理が容易になった。
    - DBスキーマ、Rubyモデル、UIのすべてにおいて一貫した用語が使用される。
- **デメリット**:
    - リモートリポジトリへの強制プッシュ（`git push --force`）と、ローカルデータベースの完全な破棄・再構築（`db:drop db:create db:migrate`）が必要となった。
    - 以前の `Facility` ベースのブランチを Pull していた他の開発者に対して、環境のリセットを強いる必要がある（チーム内合意により解決）。

## 今後の展望
- 今後追加される資産や機械のマスタは、厳密に `Equipment`, `Machinery`, または `Assets` と命名し、物理拠点（`Site`）とは明確に分離して管理する。
