# ADR 008: 取引基盤における物理拠点・組織・在庫ドメインの構造定義
**設計判断の記録 (Architecture Decision Record)**

## 📅 ステータス (Status)
**承認済み (Accepted)**: 2026-04-06

## 💡 背景 (Context)
製造リソースを包括的に管理するためには、単なる「名前」の定義に留まらず、各ドメイン（人事・物流・在庫）間の境界線（Bounded Context）を物理的なデータベースレベルでどのように整理するかの指針が必要である。

## 🏹 検討したドメイン分割方針

### 1. ドメインの分離とマイグレーションの構成
- **現状**: 開発初期の検討段階では、単一の巨大なマイグレーションファイルが想定されていた。
- **課題**: 巨大なファイルでは、「人事情報の変更」が「在庫ロジックの定義」に混じり、将来の保守性が低下する恐れがある。
- **決定**: 3つの論理的なマイグレーションに分割・再編する。
    - **`PersonnelBase`**: 人・組織・配属に関する基盤。
    - **`LogisticsBase`**: 地理的拠点（Site）および保管場所（Location）に関する基盤。
    - **`InventoryFoundation`**: 在庫・資材の所有権と所在を紐付ける基盤。

### 2. 拠点名称のグローバル標準準拠 (Why "Site"?)
ADR 002 にて最終決定された `Site` という呼称に至るまでの詳細な技術的背景。
- **対抗案としての `Facility`**: 直感的だが、製造現場では「設備自体の性能」や「施設メンテナンス」を指す用途として `Facility` が多用される。
- **選定した `Site`**: 物理的な土地や拠点を指す概念として、**SAP** や **Oracle** などのグローバルERPシステムでも「物理的な事業拠点」を指す標準的な概念（Site/Plant）として定義されている。将来の多国籍展開において、翻訳やデータ連携の齟齬を最小化するための攻めの命名である。

### 3. ハイブリッド在庫管理の明示的定義
在庫テーブル（`Inventories`）における、従来の「場所 ＝ 所有者」という固定観念の打破。
- **決定**: 「論理的な管理責任（`OrgUnit`）」と「物理的な保管場所（`Location`）」をDBカラムレベルで完全に切り離す。
- **メリット**: 同一の場所（`Location`）において、複数部門（`OrgUnit`）が在庫を混在して所持できるだけでなく、委託在庫（預かり品）などの複雑な所有権管理を、標準的なテーブル構造のみで表現可能にする。

## ✅ 決定事項 (Decisions)
1. **最初から「ドメイン分離」されたスキーマ構築**: 実装開始時点から、人事・物流・在庫を別々のマイグレーションとモデル群として構成する。
2. **Site モデルへの実務属性の初期実装**: 納品指示書や請求書等の帳票業務に耐えうるよう、住所（`postal_code`）や連絡先（`phone_number`, `fax_number`）を Site モデルに必須属性として定義する。
3. **ユーザー・タクソノミー（識別体系）の早期確立**: `Users` テーブルに最初から `user_category` と `employment_type` を設け、人間・外部・自動化エージェントを認証基盤の核で識別する。

## 📈 期待される効果 (Consequences)
- **資産管理機能への道筋**: 今後追加される設備や機械のマスタを、`Site`（場所）と衝突することなく `Equipment` や `Assets` として純粋に定義できる。
- **保守性の向上**: 特定の業務ドメイン（例：在庫管理ロジック）の変更が、人事や拠点といった土台となるドメインへ波及しにくくなる。
- **一貫性**: DBスキーマ、Rubyモデル、UIのすべてにおいて最初から一貫したグローバル標準の用語（Site, Location, OrgUnit）が使用される。
