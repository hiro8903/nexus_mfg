# ADR 007: 品目マスタの統合とBOM（部品表）アーキテクチャの導入

**日付**: 2026-04-01
**ステータス**: Accepted

## Context (背景と課題)

当社（仮想受託加工業）のシステム設計において、これまでは以下の2つのマスタを分離して管理する方針をとっていました。
1. **商品（Product）**: お客様からの預かり品、半商品、完成品など受託加工の対象フロー
2. **品番（Item）**: 仕入先から買う原材料や消耗品など、自社で用意する部品フロー

しかし、製造フローの以下の複雑な業務パターンを考慮する必要があることにしました。
- 業者への発注は「商品Aセット」で行うが、実際の入荷時には「主剤」と「副剤」に分かれ、それぞれ異なるロット・有効期限で在庫管理される。
- 自社内で購入した原材料を調合して「内製中間材（コーティング液など）」を作成し、それを最終工程で消費する。

現在の分離されたデータモデルと単一階層のレシピ（商品構成）の仕組みでは、「品番と品番を混ぜて新しい品番を作る（調合）」という概念や、「セット品をバラして在庫する（展開）」といった親子階層の表現が困難であり、システムの拡張性が阻害される懸念が生じました。

## Decision (決定事項)

1. **品目マスタへの統合 (Single Item Model)**
   - `商品(Product)` と `品番(Item)` という言葉でのテーブル分離を廃止し、すべて単一の **`品目マスタ (items)`** テーブルに統合します。
   - レコードの振る舞いを変えるために `品目区分 (item_type)` （例: 1:受託基材, 2:受託完成品, 3:購入セット品, 4:購入原材料, 5:内製中間材...）を導入し、役割を区別します。

2. **品目構成マスタ (Item BOM) の導入**
   - 既存の「標準レシピマスタ」を廃止し、自己参照テーブルである **`品目構成マスタ (item_boms)`** を導入します。
   - `parent_item_id` (作られるもの) に対して、複数の `child_item_id` (消費されるもの) を持たせることで、「何が何で出来ているか」の階層（部品表 = Bill of Materials）を無制限に表現できるようにします。

3. **現場用語の統一**
   - データベース・ソースコード上の実体は統合しますが、現場のUIメニュー名としては共通して「**品目マスタ**」とし、画面内のタブや検索フィルタで「購入品」「受託品」といった絞り込みができるようにUIを設計します。

## Consequences (結果・影響)

### メリット (Pros)
- **無限の構成階層への対応**: 半製品、調合品、セット発注、外注加工など、複雑に合流・分岐する生産プロセスを1つのデータモデルで破綻なく表現できます。
- **ロジックの共通化**: ロットの追跡（トレースバック/トレースフォワード）や在庫の引き落とし計算、原価計算などのプログラムが「品目マスタ」単一に対する処理としてきれいに1本化され、DRY原則を保てます。

### デメリット (Cons)
- **マスタ登録の難易度上昇**: 物理的な在庫単位と、発注や預かりの単位が異なる場合など、単一の品目登録だけでは完結しない「親子関係（BOM）」の定義が必須となり、管理コストが増加します。
  - *具体例1: 購入品（セット発注・バラ納品）*: 発注用の「Aセット（親）」と、実際のロット・期限管理を行うための「主剤・副剤（子）」を別々に登録し、構成表で紐づける必要がある。
  - *具体例2: 預かり品（一括・部品別管理）*: 「ユニットX（親）」として預かるが、社内ではバラして「部品a・b（子）」として別々に加工・管理する場合、構成表で分解を定義する必要がある。
  - *具体例3: 内製材（調合プロセス）*: 複数の原材料を混ぜて作った「コーティング液Z（親）」を一旦在庫する場合、そのレシピ（子たち）を構成表に定義し、在庫の親子引き落としを行う必要がある。
  - *補足*: 単純な「1対1」の物（買ったものをそのまま使う等）は品目登録のみで済むが、上記のような複雑なケースを全て「品目構成」という統一の仕組みで解決するため、初期設定には丁寧な設計が求められる。
- **システム構成用語の変更**: これまで「商品」「品番」と呼んでいたものを「品目」に頭を切り替える必要があります。それに合わせたER図、DBML、用語集の全面修正が必要になります。

## Key Implementation Rules (システム実装上の重要ルール)

今回の品目マスタ統合とBOM構成の導入を実際にシステムとして動かす際、以下の挙動を実装に盛り込む必要があります。

1. **品目区分（item_type）による動的なUIフィルタリング**
   - テーブルは1つに統合するが、入力ミスを防ぐため、特定の画面では特定の品目区分のみを初期表示・検索可能とする（例：発注画面では「購入品系」のみ、受注画面では「完成品系」のみ）。

2. **「購入セット品」の入荷展開ロジック**
   - 発注データに `購入セット品` 区分が指定されている場合、入荷検品（Receipt）処理の開始時に、自動的にBOMを参照して `子品目（原材料）` の明細行へと展開して表示する機能を実装する。

3. **「品目の一意性」の維持（共通品の統合）**
   - 実質的に同一の物理的資材（例：メーカー型番が同一の副剤）は、複数のセット商品に含まれていても必ず「単一の品目レコード」として登録・運用する。これにより、在庫の自動合算と一元管理を実現する。

4. **ロケーション（保管場所）を介した在庫の内訳管理**
   - 同じ品目であっても、どの製品（親）の隣に置いてあるか等の内訳は、品目IDではなく `ロケーション（棚番号等）` の情報を付加することで識別・計上可能とする。

5. **マルチレベルBOMの解決ロジック**
   - 内製中間材（調合液など）を含む多階層の構成において、最終的な原材料まで遡って所要量を計算できるデータアクセス（再帰的な参照）を考慮した実装とする。
