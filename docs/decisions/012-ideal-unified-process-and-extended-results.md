# ADR 012: [Ideal Concept] 検査・出荷工程の「工程マスタ」への統合と実績データの拡張方針

## ステータス
**提案中 (Proposed)**: 2026-04-07
※ 本件はフェーズ2以降の「理想系」を見据えた構想であり、実装時に要件に応じて変更される可能性があります。

## 背景 (Context)
標準的な製造業のフローには、加工（プレス、切削等）の他に、検査、梱包、発送といった多くの「作業」が含まれる。
これらを独立した機能テーブルとして個別に設計すると、以下の課題が生じる懸念があった。
- 現場の総作業時間（工数）や原価の計算ロジックが、加工とそれ以外で分散してしまう。
- 「誰が、いつ、何を行ったか」というタイムライン（トレーサビリティ）が、複数のテーブルに分断され追跡が困難になる。

## 検討した選択肢
- **案A (機能別分離)**: `inspections`（検査データ）、`shipments`（出荷データ）をそれぞれ完全に独立した機能テーブルとして作成する。
- **案B (工程統合モデル)**: すべてを「工程（Process）」の一種と捉え、同一の「実績（Work Result）」フレームワークで扱う。

## 決定事項 (Decision) - [Ideal Concept]
**案B (工程統合モデル)** を採用する。すべての現場活動を `process_masters` における一工程として定義し、以下の構造で管理する。

### 1. 「工程」としての共通管理
検査、梱包、出荷作業をすべて `process_masters` に登録し、`work_orders`（指示）と `work_results`（実績）の仕組みをそのまま利用する。これにより、誰がどの作業を何分行ったかという記録が、加工工程と全く同じ粒度で蓄積される。

### 2. 「実績」に対する詳細情報の拡張（Extension Tables）
`work_results`（作業実績）を親とし、特定の工程タイプのみ必要となる詳細情報を外部テーブルで拡張する。
- **検査工程の場合**: `inspection_results` を紐付け、物理的な計測値や合否判定を記録。
- **出荷工程の場合**: `shipping_details` を紐付け、運送会社や送り状番号を記録。
- **材料投入**: `work_order_inputs` を紐付け、どの材料ロットを消費したかを記録。

### 3. 一貫したトレーサビリティの実現
「入材(Receipt) → 投入(Input) → 加工(Result) → 検査(Result/Inspection) → 出荷(Result/Shipping)」という全履歴を、単一の `work_results` チェーンで追跡可能にする。

## 期待される効果 (Consequences)
- **原価計算の統合**: プログラム上で「全工程の作業時間を合計する」だけで正確な作業原価が算出できる。
- **柔軟な工程追加**: 将来「ラベル貼り」や「外部委託」などの新しい種類の作業が増えても、DBの根幹を崩さずに「工程」として追加できる。
- **実装のDRY化**: 「誰がいつ始めたか」といった作業開始・終了のUI、および認可ロジックを全工程で共通化できる。
