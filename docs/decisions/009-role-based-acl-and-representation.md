# ADR 009: 組織・役割ベースのアクセス制御（RBAC）とトランザクション表示担当者モデル

## ステータス
**承認済み (Accepted)**: 2026-04-06

## 背景 (Context)
製造業の基幹システムにおいて、権限管理は単なる「ログインの可否」ではなく、「特定の組織（営業部、製造部一課など）において、どのような役割（一般、承認者など）を担っているか」というコンテキストに依存する。

また、OJT（教育）や忙しい同僚の「代行入力」、あるいは作成後の微修正などの実務において、システム上の記録主体（作成者）と、対外的な書類上の主体（表示担当者・署名）を使い分けたいという強いニーズがある。

これらをシステムとして正確かつ柔軟に表現するため、[ADR 003](./003-permission-management.md) を拡張した「組織 × 役割」ベースの権限解決モデルと、トランザクションにおける「表示担当者」モデルを定義する。

## 🏹 決定事項 (Decision)

### 1. 組織・役割ベースの権限解決
`org_unit_permissions` テーブルに `role`（役割）カラムを追加し、特定の組織における役割ごとの権限を定義する。

- **`assignments.role`**: ユーザーがその組織に「どのような役割」で配属されているかを保持。
- **`org_unit_permissions.role`**: その組織の「その役割」に対して、どの機能（`key`）をどの強さ（`level`）で許可するかを定義。

#### 権限レベルの標準指針
- **0: Reader (閲覧)**: データの詳細を見ることができるが、保存はできない。
- **1: Editor (編集)**: データの作成・更新・削除ができる（ドラフト状態まで）。他者の作成したデータの編集も、所属組織が同じであれば可能。
- **2: Approver (承認)**: データの確定・承認・ロック解除ができる（正式な意思決定）。

### 2. トランザクションにおける「表示担当者」モデル
見積書や発注書などの伝票（Transaction）系テーブルにおいて、以下の3つのユーザーIDを保持する設計を標準とする。

- **`created_by_id`**: システム上の物理的な作成者（自動追跡、監査ログ用）。
- **`last_updated_by_id`**: 最終編集者（自動追跡）。
- **`applicant_id`**: **表示・署名上の担当者（実要求者、手動選択・事前設定・変更可能）**。書類の印字や、対外的な公式担当者として使用。

### 3. 権限解決（Effective Permission）のアルゴリズム
特定の操作（例：営業部の見積の承認）を判定する際、システムは以下のロジックで権限を「合算・継承」して解決する。

1. ユーザーの **現在有効な** すべての `assignments` を取得。
   （※ **判定基準**: `start_date <= 今日` かつ `end_date` が NULL または `今日 <= end_date`）
2. それぞれの `assignment` が持つ `(org_unit_id, role)` の組み合わせに対して定義された `org_unit_permissions` を検索。
3. **[重要：階層継承]** 指定組織に該当の権限キーが見つからない場合、その組織の **親組織（parent_id）を再帰的に遡り**、同一の `role` に対して定義された権限を検索する。
   - これにより、全社共通の役職権限を最上位組織で一度定義するだけで、全子組織の同役職者に自動適用され、登録漏れと管理コストを防止する。
4. 全配属および継承ルートから得られた `permission_level` の中の **最大値（Max）** を、そのユーザーの「実質的な有効権限」とする。

#### 権限継承の構造イメージ（ツリー形式）

```text
株式会社○○ (全社) [★ここで「課長」の権限を定義]
├── 営業部 (定義なし：親を自動参照)
│   └── 営業一課 (定義なし：親の親を自動参照)
│       └── [配属] 田中課長 (全社定義の「課長権限」を継承)
```

**【田中課長のアサインデータ例 (`assignments`)】**
| カラム名 | 値 | 意味 |
| :--- | :--- | :--- |
| **user_id** | 101 | 田中さんのID |
| **org_unit_id** | 501 | 「営業一課」のID |
| **role** | `2` | 役割：**課長** (Manager) |
| **job_title** | "営業一課 課長" | 書類印字用の役職名 |
| **start_date** | 2026-04-01 | （有効期間内であること） |

- **継承の仕組み**: システムは 田中さんの `org_unit_id (501)` を見に行き、権限がなければ親の `営業部`、さらに親の `株式会社○○` へと自動的に遡ります。そこで `role: 2 (課長)` に対する定義を見つけるため、田中さんは自動的に権限が付与されます。
- **管理のメリット**: 100の部署があっても、共通権限の登録は最上位の **「1回」** で済み、登録漏れが物理的に発生しません。

#### 役割の衝突回避と安全な継承

階層継承を安全に運用するため、以下の原則を適用する。

- **役割（Role ID）の分離**: 「全社共通の管理者(Role 2)」と「特定PJのリーダー(Role 5)」のように、役割のIDを分ける。これにより、PJリーダーが親組織（全社）の管理者権限を誤って継承するリスクを物理的に遮断する。
  ```ruby
  # [Assignment モデルでの実装想定]
  enum :role, {
    member: 0,       # 平社員（部署・PJの標準権限）
    manager: 5,      # 課長・部長（部門全体のマネジメント権限：全社継承あり）
    pj_leader: 10,   # 特定PJ内でのリーダー権限（★全社権限とは切り離す）
    evaluator: 15,   # 人事考課・評価者用の特例権限
  }
  ```
- **複数配属による合算**: ユーザーは「営業部(Member)」と「PJ-A(Leader)」の2つの配属（Assignment）を持つことができる。システムは各配属から解決された権限をすべて足し合わせる（UNION）ため、所属部署での平社員としての権利を保ちつつ、PJ内でのみ特別な権限を行使することが可能。

### 4. 行動・目的ベースの抽象組織（Activity-based OrgUnit）の活用
物理的な部署（営業部、製造部）以外に、特定の「目的」や「行動」を一つの `OrgUnit` として定義することを許容し、強力な権限移譲に活用する。

- **例: 「棚卸特別チーム」**: 全サイトの在庫を修正できる権限。一時的に選抜されたメンバーをこの「組織」にアサインすることで、本来の部署を問わず権限を付与できる。
- **例: 「内部監査チーム」**: 全機能の閲覧権限（Reader）。
- **メリット**: `assignments` の `start_date` / `end_date` を活用することで、時限的な特例権限の設定に対し、**将来の運用フェーズでソースコードの変更（デプロイ）を伴わない「マスタ登録のみ」での安全な制御** が可能となる。

## ✅ 期待される効果 (Consequences)

### メリット
- **実地の運用に強い**: 「新人が起票して先輩の名前で出す」「営業部の他人の見積も営業部員なら修正できる」といった、現場の自然な協力体制を妨げない。
- **管理の自動化**: 「課長を異動させた際、以前の部署の権限が消え、新しい部署の課長権限が自動で付与される」といった理想的な運用が、配属レコード（Assignment）の更新だけで完結する。
- **特例の吸収**: PJや委員会などの限定的な役割による一時的権限移譲を、メイン組織の構成を乱さずに吸収できる。

### デメリット
- **判定ロジックの実装**: `any?` や `max` を含む SQL（または ActiveRecord）のクエリが必要となり、単純な boolean チェックよりは実装の複雑度が増す。

## 📈 現状と今後の展望

### 1. 現状 (Current Status)
本ADRの時点では、**「権限判定のための器（データ構造）」** である `org_unit_permissions` マスタの構築にフォーカスしている。
- 具体的な認可判定のためのライブラリ（Pundit 等）や、実際の判定ロジックを記述する `app/policies/` ディレクトリはまだ存在しない。
- まずは「どのような複雑なルールも表現できるマスタ基盤」を完成させる。

### 2. 次フェーズへのロードマップ (Future Roadmap)
見積（Quotations）や発注などの「業務機能（Transactions）」を実装するフェーズにおいて、以下の手順で認可層を結合する。

- **認可層（Policy）の導入**: Rails の標準的な認可パターンである `Policy` クラスを導入する。
- **複雑なビジネスロジックの集約**: 「500万以上の見積は特定役職が必要」といったドメイン固有の条件分岐は、マスタに無理やり持たせるのではなく、Policy クラス内で Ruby のコードとして記述し、保守性と柔軟性を両立させる。
- **標準ロールのシード定義**: 営業部・製造部・事務管理などの代表的な組織タイプに対し、標準的な `role` （一般、主任、課長、部長）ごとの権限セットをシードデータとして定義し、開発初期の生産性を高める。
