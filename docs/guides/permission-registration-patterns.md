# 権限定義および運用登録パターン ガイド

このドキュメントは、組織・役割ベースの権限管理（[ADR 009](../architecture/adr/009-role-based-acl-and-representation.md)）を用いて、製造現場の多様な業務シナリオをシステムにどう登録・運用するかをまとめたガイドラインです。

> [!IMPORTANT]
> **運用の鉄則: 「ユーザーに直接権限を付けない」**
> すべての権限は「組織(OrgUnit)」と「役割(Role)」の組み合わせに対して付与します。
> ユーザーの権限を変更したい場合は、そのユーザーの「配属(Assignment)」を変更するのが正解です。

---

## 1. 標準的な部門内運用（営業部の例）

部門ごとに「一般社員」と「決裁者（課長・部長）」で操作を出し分ける、最も基本的なパターンです。

*   **業務想定**: 
    - 営業部員は全員、自分や同僚の見積を作成・編集できる。
    - 課長以上のみが、見積を「承認（確定）」できる。
    - 他部署（製造部など）は見積の内容を「閲覧」だけできる。

#### 📌 マスタ登録の手順
1. **組織(OrgUnit)を登録**: `営業部`, `製造部`
2. **権限(OrgUnitPermissions)を登録**:
   - `営業部` × `一般(Member)` × `quotation` ＝ **レベル 1: Editor(編集)**
   - `営業部` × `課長(Manager)` × `quotation` ＝ **レベル 2: Approver(承認)**
   - `製造部` × `一般(Member)` × `quotation` ＝ **レベル 0: Reader(閲覧)**
3. **配属(Assignment)を登録**:
   - ユーザーAを `営業部` の `一般` に配属 → 自動的に「見積編集」が可能に。
   - ユーザーBを `営業部` の `課長` に配属 → 自動的に「見積承認」が可能に。

## 2. 行動・ミッションベースの特例運用（棚卸チームの例）

「部署」という枠組みを超えて、特定のプロジェクトや期間限定の「行動」に対して権限を与えるパターンです。

*   **業務想定**: 
    - 年2回の大型棚卸のため、各部署から選抜された「棚卸リーダー」に対し、全工場の在庫データ修正権限を一時的に与えたい。
    - 普段、そのメンバーは自分の部署の在庫しか触れない。

#### 📌 マスタ登録の手順
1. **仮想組織(OrgUnit)を登録**: `2026年春期棚卸チーム` (区分: `2:Committee`)
2. **権限(OrgUnitPermissions)を登録**:
   - `棚卸チーム` × `リーダー` × `inventory` ＝ **レベル 2: Approver(修正・確定)**
3. **配属(Assignment)を登録**:
   - 選抜メンバーを `棚卸チーム` の `リーダー` として **兼務** 登録する。
   - **ポイント**: `end_date`（離任日）を棚卸終了日に設定しておくことで、期間終了後に自動で権限が消滅します。

## 3. OJTおよび代理入力（新人教育の例）

システム上の「入力した人」と、書類上の「責任者（署名）」を使い分けるパターンです。

*   **業務想定**: 
    - 新人Aさんが見積書を作成（タイピング）するが、お客様に出す書類の署名欄（担当者名）には、指導役の先輩Bさんの名前を載せたい。
    - 後日、内容に不備があった際、システム上で「実際に誰が打ち込んだか」を確認できるようにしたい。

#### 📌 データの登録状態（見積レコード）
1. **`created_by_id`**: `新人A` (システムが自動で固定。教育実績の証跡になる)
2. **`represented_by_id`**: `先輩B` (入力時に選択。書類に印字される「顔」となる)
3. **`last_updated_by_id`**: `先輩B` (先輩が新人の内容をチェックして微修正した際、ここに記録される)

## 4. 全社レベル・階層を超えた管理運用（階層継承モデル）

物理的な拠点に縛られず、会社全体を俯瞰する定義を効率的に行うパターンです。

#### 📌 権限継承の構造イメージ
```text
株式会社○○ (全社) [★ここで「課長」の権限を定義]
├── 営業部 (定義なし：親を自動参照)
│   └── 営業一課 (定義なし：親の親を自動参照)
│       └── [配属] 田中課長 (全社定義の「課長権限」を自動継承)
```

**【田中課長のアサインデータ例】**
| user_id | org_unit_id | role | job_title |
| :--- | :--- | :--- | :--- |
| 101 | 501 (営業一課) | `manager` | "営業一課 課長" |

- **メリット**: 全社共通の役職権限は、トップ組織に一度登録するだけで、全部署の同役職者に「染み出す」ように自動適用されます。100の部署があっても登録は1回で済みます。

## 5. 安全な役割の使い分け（特権昇格の防止）

「プロジェクトリーダー」が誤って「全社の管理者権限」を引き継がないよう、役割の種類（Role ID）を使い分ける運用を徹底します。

#### 📌 システムで定義されている役割（MECEなブロック運用）

衝突を防ぎ、将来の拡張性を確保するために **100単位のブロック** で採番します。これにより、人間・外部・自動化の全領域を網羅します。

| ブロック | ID 範囲 | 継承 | 主な利用シーン |
| :--- | :--- | :--- | :--- |
| **A: 標準組織** | 0 - 99 | ◯ | 営業、製造、総務などの階層組織（正社員用） |
| **B: PJ・委員会** | 100 - 199 | × | 社内プロジェクト、タスクフォース、委員会 |
| **C: 専門職能** | 200 - 299 | × | 内部監査、人事考課、棚卸などの特殊任務 |
| **D: 外部協力者** | **300 - 399** | **隔離** | **派遣、外注先、ベンダー（社内権限と完全分離）** |
| **E: サービス** | **800 - 899** | **自動** | **API連携、IoT機器、自動バッチ（非人間）** |
| **F: システム** | 900 - 999 | 特権 | システム管理者、保守エンジニア |

```ruby
# [システム実装の裏側：Assignment モデルでの Enum 定義の理想的な形]
enum :role, {
  # --- 0番台: 標準的な階層組織用 (全社継承を推奨) ---
  member: 0,       # 平社員：部署の標準的な操作
  leader: 10,      # チームリーダー・主任
  manager: 20,     # 課長・部長：部署全体の管理権限（全社染み出しに最適）

  # --- 100番台: 特定のプロジェクト・委員会用 (局所完結) ---
  pj_member: 100,  # プロジェクトの一般参加者
  pj_leader: 110,  # プロジェクト責任者：PJ内の特権

  # --- 200番台: 専門職能・横断的な活動用 (用途限定) ---
  evaluator: 200,  # 評価者：人事考課期間の評価スコア操作用
  auditor: 210,    # 内部監査人：全拠点の「閲覧のみ」
  stocktaker: 220, # 棚卸担当：在庫データの強制書き換え用

  # --- 300番台: 外部協力者・サードパーティ (社内権限から完全分離) ---
  contractor: 300, # 派遣・業務委託：限定された機能のみアクセス
  vendor: 310,     # ベンダー：保守やデータ納品用
  supplier: 320,   # サプライヤー：将来的には portal 連携を想定

  # --- 800番台: システムアカウント・サービス (非人間) ---
  api_service: 800,# API連携用：外部システムからのデータ取得・反映
  iot_device: 810, # IoT機器：センサー等の実績データ自動投入用
  batch_bot: 820,  # 自動バッチ：夜間の在庫計算などのシステム処理

  # --- 900番台: システム管理用 ---
  admin: 999       # システム管理者：全機能のフルアクセス
}
```

#### 📌 「営業部員」と「PJリーダー」の兼務例
田中さんが営業部では一般社員、プロジェクトAではリーダーを務める場合、**2つの配属 (Assignment)** を登録します。
1.  `営業部` × `role: member (0)` ＝ 営業部の一般権限をゲット
2.  `PJ-A` × `role: pj_leader (110)` ＝ プロジェクト内でのみ特権をゲット

システムはこれらを足し合わせるため、田中さんは「営業部では平社員、でもプロジェクト内では 100番台の特権を持つ」という、セキュアな状態になります。

#### 💡 コラム：権限はどう細分化されるのか？（3要素の掛け合わせ）

「営業部長」と「製造部長」のように、特定の権能を細かく分けて管理したい場合、**Enum を増やす必要はありません。** 以下の 3 要素を「マスタ登録」で組み合わせることで、無限のパターンを表現できます。

1.  **組織 (OrgUnit)** : どこ（営業部? 製造部?）
2.  **役割 (Role)** : どのランク（manager? leader? member?）
3.  **職能 (Permission Key)** : 何を（見積? 在庫? 発注? 人事考課?）

**【設定例】**
- **営業部長の権限**: `(営業部)` ＋ `(manager)` ＋ `(quotation)` ＝ レベル2
- **製造部長の権限**: `(製造部)` ＋ `(manager)` ＋ `(inventory)` ＝ レベル2

このように細分化することで、同じ `manager` という Enum を使いつつも、**「営業部長が勝手に他部署の在庫を調整する」といった事故を物理的に防ぐ** ことが可能です。

## 💡 実務上のFAQ（運用・保守）

### Q. 「課長」は「一般社員」の権限も持っていますか？
**A. はい、持っています。**
権限レベル（Level）は階層構造になっています。`2:Approver` は `1:Editor` および `0:Reader` の権限を含んでいるため、課長は一般社員ができる「編集」も当然可能です。

### Q. 1人のユーザーが複数の部署で異なるレベルの権能（例：片方は編集、片方は閲覧）を持ったら？
**A. 「高い方のレベル」が優先されます。**
例えば、A部で「閲覧(0)」、B部で「編集(1)」の権限を持っていた場合、その機能に対しては「編集(1)」として振舞います。これにより、「兼務しているのに片方の部署の制約で操作できない」というストレスを防ぎます。

### Q. 組織図にない「プロジェクト」も登録していいですか？
**A. 大歓迎です。**
`org_type` に `PJ(プロジェクト)` や `Committee(委員会)` が用意されています。物理的な階層組織（ピラミッド）に縛られず、マトリクス組織やタスクフォースを積極的に `OrgUnit` として登録し、そこに権限を紐付けてください。
