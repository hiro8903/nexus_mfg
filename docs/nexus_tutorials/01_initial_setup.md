# NexusMfg チュートリアル：第1章 プロジェクトの開始とGitHub連携

本チュートリアルでは、Rails 8 を使用した高度な製造管理システム「NexusMfg」をゼロから構築する手順を解説します。

## 0. 本章の目的と学習内容

本章では、開発をスタートするための「土台作り」を行います。以下の手順を通じて、一切の無駄がないクリーンなプロジェクト基盤を構築します。

1. **ワークスペースの作成**: コードを格納する作業ディレクトリを作ります。
2. **Railsアプリの生成**: Tailwind CSS や Importmap を統合したモダンな基本構成のアプリを生成します。
3. **リモートリポジトリの接続**: コードの履歴管理を開始し、GitHub への初回アップロード（Push）を完了させます。

※ プロジェクト全体の完成像についてはプロジェクト直下の `README.md` を、チュートリアル全体の目次や進行状況については [チュートリアルポータル（目次）](./00_tutorial_index.md) を参照してください。

---

## 1. ワークスペース（作業ディレクトリ）の準備

まずは、アプリケーションのソースコード群を格納するための土台となるディレクトリを作成し、そこへ作業場所を移します。実務ではターミナル上で「自分が今どのパス（場所）にいるか（＝カレントディレクトリ）」を常に意識することが重要です。

- **実行ディレクトリ**: 任意のディレクトリ（例：`~/Desktop` や `~/work` など）
- **作業ブランチ**: なし (Git管理前)
- **コマンド**:
```bash
mkdir environment
cd environment
```

### コマンドの解説
- `mkdir environment`: `environment` という名前の新しいディレクトリを作成します。<a id="ref-1"></a>[※1](#appendix-1)
- `cd environment`: 作成したディレクトリの中に、ターミナルのカレントディレクトリを移動させます。<a id="ref-2"></a>[※2](#appendix-2)

### ✅ 次へ進むための確認事項
- ターミナル上でエラーが出ず、カレントディレクトリが `environment` の中に移動していること。

## 2. アプリケーションの生成

次に、作成したディレクトリの中で Rails の基本プロジェクトを生成します。

- **実行ディレクトリ**: `environment` （手順1で移動した場所）
- **作業ブランチ**: なし (Git管理前)
- **コマンド**:
```bash
rails new nexus_mfg --css tailwind --javascript importmap
```

### コマンドの解説
- `nexus_mfg`: カレントディレクトリ直下にこの名前のディレクトリが作成され、その中にRailsアプリケーションのスケルトン（雛形のソースコード群）が一式展開されます。
- `--css tailwind`: フロントエンドのスタイリングに、モダンなユーティリティファーストCSSである Tailwind CSS を統合します。<a id="ref-3"></a>[※3](#appendix-3)
- `--javascript importmap`: Rails 8 の標準機能である Importmap を使い、軽量にJavaScriptを管理・配信します。<a id="ref-4"></a>[※4](#appendix-4)

### ✅ 次へ進むための確認事項
- コマンド実行後、`environment` ディレクトリ内に `nexus_mfg` フォルダが作成され、中に大量のファイルが自動生成されていること。

## 3. リモートリポジトリ（GitHub）への接続

作成したコードを世界（GitHub）へ届けます。

### ステップ1：作業ディレクトリへの移動と初回コミット
生成されたアプリのフォルダへ移動し、Gitの履歴追跡を開始します。

- **実行ディレクトリ**: `environment`
- **作業ブランチ**: なし (Git管理前)
- **コマンド**:
```bash
cd nexus_mfg
git init
git add .
git commit -m "残したいメッセージ"
```

### コマンドの解説
- `cd nexus_mfg`: 生成されたアプリケーション本体のフォルダへ移動します。今後の作業はすべてこの「プロジェクトルート」で行います。（※ `cd` の意味については [※2](#appendix-2) を参照）
- `git init`: このディレクトリを、Git（ソースコードのバージョン管理システム）のローカルリポジトリとして初期化し、履歴追跡を開始します。
- `git add .`: カレントディレクトリ配下の全ての変更を、コミットの準備段階であるステージングエリアに引き上げます（Trackする）。<a id="ref-5"></a>[※5](#appendix-5)
- `git commit -m "残したいメッセージ"`: ステージングされたファイル群を、1つの「歴史の記録（コミット）」としてローカル環境に確定・保存します。<a id="ref-6"></a>[※6](#appendix-6)

### ✅ 次へ進むための確認事項
- エラーが出ずにコマンドが完了すること。（※この時点からローカルのGitが有効になり、ブランチ名が `main` または `master` として確定期に入ります）

### ステップ2：リモートリポジトリの紐付け
ローカルに作成した歴史（コミット）をインターネット上の保管庫であるGitHubへリンクさせます。
（`<your-github-username>` はご自身のユーザー名に置き換えてください）

- **実行ディレクトリ**: `nexus_mfg`（プロジェクトルート）
- **作業ブランチ**: `main` (または `master`)
- **コマンド**:
```bash
git remote add origin https://github.com/<your-github-username>/nexus_mfg.git
```

### コマンドの解説
- `git remote add origin ...`: ローカルリポジトリに対し、アップロード先となる外部（リモート）リポジトリのURLを「origin」という名前で登録します。<a id="ref-7"></a>[※7](#appendix-7)

### ✅ 次へ進むための確認事項
- ターミナルでエラーが出ないこと。（本当に紐付いたか確認したい場合は `git remote -v` と打ち込み、URLが表示されれば成功です）

### ステップ3：初回のプッシュ

- **実行ディレクトリ**: `nexus_mfg`（プロジェクトルート）
- **作業ブランチ**: `main` (または `master`)
- **コマンド**:
```bash
git push -u origin main
```

### コマンドの解説
- `git push -u origin main`: 手元の `main` ブランチの歴史を、登録済みリモートである `origin` に対して送信（プッシュ）します。 `-u` によって次からブランチ指定を省略可能にします。<a id="ref-8"></a>[※8](#appendix-8)

### ✅ 次へ進むための確認事項
- ご自身のGitHub画面をブラウザで更新し、NexusMfg のコード群が正しくアップロードできていることが確認できること。

## 🎯 Appendix: 初心者向け用語・概念解説

<a id="appendix-1"></a>
**[※1 Make Directory (mkdir)](#ref-1)**
ディレクトリとはWindowsやMacでいう「フォルダ」のことです。今回指定した `environment` は複数のアプリをまとめるためのただの「箱」なので、名前は自分の好きなようにつけて構いません。

<a id="appendix-2"></a>
**[※2 Change Directory (cd)](#ref-2)**
ターミナル上で「自分が今どのパス（場所）にいるか」を表現する場所を「カレントディレクトリ」と呼びます。これ以降に叩くコマンドは、すべて移動した先のディレクトリ内部で実行・配置されることになります。

<a id="appendix-3"></a>
**[※3 `--css` オプションと Tailwind CSS](#ref-3)**
> **Tailwind CSS とは？**: 「ボタンを青くする」「文字を大きくする」といったデザインの指示を、HTMLの中に直接短い単語で書き込んでいける非常に便利な道具です。別のCSSファイルと行ったり来たりする手間が省けるため、プロの現場でも大流行しています。
> 
> **豆知識（オプションの仕組み）**: アプリという家を建てる際の「設備オプション」のようなものです。
> - **別の選択肢**: `bootstrap`（昔からある有名な枠組み）などを指定できます（例: `--css bootstrap`）。
> - **もしつけなかったら？**: Railsは特定のデザインセットを用意してくれません。完全に"まっさらなCSS"を自力で1から書くことになります。
> - **どうやって調べるの？**: この書き方はTailwindではなく **Railsの機能** です。一番確実なのはターミナルで `rails new --help` コマンドを実行することです。以下のように、Rails自身が「CSSはこれを指定できるよ」と教えてくれます。
>   ```text
>   -c, [--css=CSS]    # Choose CSS processor. 
>                      # Possible values: tailwind, bootstrap, bulma, postcss, sass
>   ```
> 
> 👉 [公式リファレンス (Resource Hub) へ飛ぶ](./resource_hub.md#tech-tailwind)

<a id="appendix-4"></a>
**[※4 `--javascript` オプションと Importmap](#ref-4)**
> **Importmap とは？**: 従来はJavaScriptを動かすために複雑な下準備（ビルド等の環境構築）が必要でしたが、これを使うと面倒な設定をすべてすっ飛ばして、サクッとブラウザで動かせるようになる画期的な仕組みです。
> 
> **豆知識（オプションの仕組み）**:
> - **別の選択肢**: Reactなどのより高度な技術を使いたい場合は、`esbuild` などの強力な構築セットを指定します（例: `--javascript esbuild`）。
> - **もしつけなかったら？**: 実は Rails 8 では、何も指定しないと勝手にこの `importmap` （標準設定）が選ばれます。ただ、実務では「このアプリはどの方式を選んで始めたか」を明示するためにオプションとして書き記すことがよくあります。
> - **どうやって調べるの？**: こちらも `rails new --help` で確認可能です。
>   ```text
>   -j, [--javascript=JAVASCRIPT]   # Choose JavaScript approach. Default: importmap
>                                   # Possible values: importmap, bun, webpack, esbuild, rollup
>   ```
> 
> 👉 [公式リファレンス (Resource Hub) へ飛ぶ](./resource_hub.md#tech-importmap)

<a id="appendix-5"></a>
**[※5 ステージング (Staging / Track)](#ref-5)**
作成したファイルをいきなり歴史に刻む（コミットする）のではなく、「このファイルたちを次の記録対象にするぞ」とリストアップする「仮置き場」のような概念です。今回は `add .`（ドット）を使うことで、ディレクトリ内の全ての変更を一度に仮置き場へ引き上げています。
👉 [公式リファレンス (Resource Hub) へ飛ぶ](./resource_hub.md#tech-git)

<a id="appendix-6"></a>
**[※6 `git commit` の `-m` オプション（見出し）](#ref-6)**
Gitで歴史を記録する際、「この時に何を変更したか」という見出し（Message）を残す必要があります。`-m` はそのメッセージを、コマンドラインから直接入力するためのオプションです。
> **もし `-m` をつけなかったら？**: `git commit` とだけ打つと、タイトルを入力するためにターミナル上で強制的に Vim などのテキストエディタが起動します。初心者はここで「元の画面への戻り方がわからない！」とパニックになりやすいため、最初は `-m` を使ってサクッと見出しを残すのが安全です。
> 
> 👉 [公式リファレンス (Resource Hub) へ飛ぶ](./resource_hub.md#tech-git)

<a id="appendix-7"></a>
**[※7 リモートリポジトリ (Remote) と Origin](#ref-7)**
Gitは自分のパソコン（ローカル）だけでも履歴を持てますが、チーム開発やPC故障時のバックアップのために、インターネット側（GitHubなど）にも履歴の保管庫を置きます。これをリモートと呼びます。「origin（起源・原点）」という単語は、メインのリモート保管庫に対してつける世界共通の慣習（ただのあだ名）です。
👉 [公式リファレンス (Resource Hub) へ飛ぶ](./resource_hub.md#tech-github)

<a id="appendix-8"></a>
**[※8 Push（プッシュ）と `-u`（Upstream）](#ref-8)**
プッシュとは、自分の手元の歴史データをインターネット上のリモート保管庫へ「押し出す」アクションのことです。`-u` (Upstreamの略) を初回に指定することで、「このローカルのmainは、常にGitHubのmainに対応しているんだな」とGitが記憶してくれます。これにより、次回からはただ `git push` と打つだけで済むようになります。
👉 [公式リファレンス (Resource Hub) へ飛ぶ](./resource_hub.md#tech-git)
